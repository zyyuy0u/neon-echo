# 《霓虹殘響 NEON ECHO》完整開發計畫書

> 版本：v1.0（2026-07-17）｜狀態：**待 user 確認**
> 一款跑在瀏覽器的第三人稱 3D 開放世界純探索遊戲，合成波霓虹美術，單人、電腦鍵鼠操作。

---

## 1. 產品願景

### 1.1 一句話定義
你在一座被時間凍結的霓虹之城醒來，靠逐步解鎖的移動能力抵達越來越遠的奇觀地標，
收集散落的「殘響」拼出這座城市沉寂的真相。

### 1.2 三大設計支柱（所有決策的裁決依據）
1. **移動本身就是樂趣**——跑、跳、衝刺、滑翔的手感是整個遊戲的地基，寧可地圖小也不容忍手感差。
2. **「看到就想去」**——薩爾達式視覺動線設計：每個抬頭可見的地標都到得了，每次繞路都有回報。
3. **少即是精**——小規模開放世界（約 800m × 800m），每一塊區域都手工打磨，不灌水。

### 1.3 目標體驗規格
- 單周目 45–90 分鐘；完美收集 2–3 小時。
- 無戰鬥、無死亡懲罰（墜落深淵僅回到最近落點）。
- 情緒基調：孤獨但不恐怖，寧靜的壯麗，最後是釋然。

## 2. 遊戲設計

### 2.1 世界觀與敘事（環境敘事，無過場動畫）
**背景**：城市 AI「AURORA」在一場資訊風暴來臨前，將全城居民的意識上傳為「殘響」封存，
城市因此靜止。玩家是風暴過後第一個醒來的「回聲行者」。
**敘事載體**：12 座全息碑文（Memory Steles）散落各區，逐段揭露事件；
收集殘響碎片達門檻後，中央核心開啟 → 結局選擇：**喚醒城市** 或 **讓殘響安眠**（雙結局）。
碑文全文中英雙語，介面字幕化呈現。

### 2.2 地圖結構（800m × 800m，中央輻射四區）
```
        北｜深谷燈海 Lumen Chasm（終局區，需全能力）
西｜倒懸環        【靜滯廣場】        東｜霓虹巨塔
   Inverted Ring   （出生點/教學）      The Spire
        南｜廢棄天梯 Sky Lift（第一個目標區）
```
| 區域 | 地標奇觀 | 進入所需能力 | 主題 |
|---|---|---|---|
| 靜滯廣場（中央） | 休眠核心 | 無（教學動線） | 甦醒、基礎移動教學 |
| 南・廢棄天梯 | 斷裂的天空電梯 | 無 → 內部深處需衝刺 | 垂直入門、第一座聖所 |
| 東・霓虹巨塔 | 全圖最高塔，塔頂可俯瞰全圖 | 衝刺 | 攀爬挑戰、第二座聖所 |
| 西・倒懸環 | 反重力懸浮巨環 | 二段跳 | 浮空平台、第三座聖所 |
| 北・深谷燈海 | 發光深谷 + 城市核心 | 衝刺+二段跳+滑翔 | 終局、結局裝置 |

設計規則：任一地標在地圖 60% 以上的位置抬頭可見（fog 之外用剪影+發光呈現）；
每條主要動線上每 30–45 秒必有一個「可互動物或視覺事件」。

### 2.3 移動系統手感規格（M1 起始值，實測後調參）
| 參數 | 初始值 | 備註 |
|---|---|---|
| 跑速 / 空中控制 | 8 m/s / 0.65 | 加速 0.15s 到頂，止滑 0.1s |
| 跳躍高度 | 2.2 m | 可變跳高（提早放開較低） |
| Coyote time / 跳躍緩衝 | 120 ms / 100 ms | 手感的靈魂參數 |
| 衝刺（解鎖1） | 20 m/s × 0.18s，冷卻 0.8s | 空中可用，FOV 輕微外擴 |
| 二段跳（解鎖2） | 高度 1.8 m | 重置空中衝刺 |
| 滑翔（解鎖3） | 下降 2.5 m/s，前進 10 m/s | 上升氣流區可爬升 |
| 相機 | 追尾距離 5.5 m，碰撞探測，靈敏度可調 | 滑鼠控制，可反轉軸 |
所有參數集中於單一 `tuning.ts` 設定檔 + 開發模式即時調參面板（手感迭代效率的關鍵）。

### 2.4 能力解鎖進程（銀河惡魔城式門鎖）
```
出生（跑/跳）→ 南區聖所①取得【衝刺】→ 東區聖所②（需衝刺）取得【二段跳】
→ 西區聖所③（需二段跳）取得【滑翔】→ 北區核心（需全能力）→ 結局
```
硬性工程要求：**可達性以圖模型建模並寫成自動化測試**——
「每個收集物/聖所在其前置能力下必可達、且不可提前抵達關鍵鎖區」是機器可驗證的驗收條件。

### 2.5 收集與謎題
- **殘響碎片 × 40**：發光收集物，站上即收；30 枚開啟核心。配置密度遵循 2.2 的 30–45 秒規則。
- **迴路聖所 × 3**（小型空間解謎房，各引入一個機制並頒發能力）：
  - ①「脈衝軌道」：定時移動平台 + 節奏跳躍 → 頒發衝刺
  - ②「光橋迴路」：踩踏開關以正確順序點亮光橋 → 頒發二段跳
  - ③「風之井」：上升氣流 + 滑翔原型體驗 → 頒發滑翔
- **記憶碑文 × 12**：互動後顯示雙語敘事文本，地圖上不標示（鼓勵探索），收齊有隱藏結語。

### 2.6 UI / UX
- 主選單、暫停選單、設定（音量/靈敏度/語言/減少動態/按鍵重映射）、收集進度環。
- HUD 極簡：能力圖示 + 碎片計數，探索時自動隱藏。
- 存檔：localStorage 自動存檔（收集/能力/位置），支援手動重置。

## 3. 美術與音訊方向

### 3.1 合成波霓虹設計系統
- **色板**（統一管理於 `palette.ts`）：
  夜空深紫 `#1a0b2e` ／ 地平線日落 `#ff6b35`→`#c73866` 漸層 ／
  主霓虹洋紅 `#ff2ec4` ／ 輔助青 `#00e5ff` ／ 警示黃 `#ffd319` ／ 結構暗藍 `#16213e`
- **手法**：低多邊形幾何 + 自發光（emissive）霓虹描邊 + Bloom 泛光（美術的命脈）+
  距離霧（藏繪製距離兼營造氛圍）+ 網格地面元素 + 緩慢移動的體積感天空。
- 效能與質感的取捨原則：幾何簡單、光影講究。不用即時陰影全開，改用烘焙感假陰影 + 少量關鍵動態光。

### 3.2 角色方案
CC0 授權風格化人形（Quaternius / Kenney 素材庫優先）+ 霓虹重製材質；
動畫用素材內建 clip（跑/跳/滑翔）+ 程序化補間。建立 `ASSETS.md` 記錄每個資產的來源與授權。
若 CC0 模型皆不合用 → 降級方案：發光幾何構成的「光構體」角色（純程序化，免動畫資產，風格反而獨特）。

### 3.3 音訊
- 音樂：CC0/CC-BY synthwave 環境音樂（來源與授權記入 `ASSETS.md`），分區域漸變。
- 音效：Web Audio 合成為主（跳躍/收集/解鎖提示音走合成器風格，与美術一致）。

## 4. 技術架構

### 4.1 技術棧
| 層 | 選擇 | 理由 |
|---|---|---|
| 語言/建置 | TypeScript (strict) + Vite | 型別安全、極速迭代 |
| 渲染 | Three.js（原生） | 遊戲以幀循環為中心，原生架構最乾淨、效能天花板最高 |
| 物理 | Rapier (`@dimforge/rapier3d-compat`, WASM) | 角色控制器、碰撞、最佳 Web 物理效能 |
| 後製 | `postprocessing` | Bloom / 色調映射 / 暈影 |
| UI 層 | 原生 DOM + CSS（覆蓋於 canvas 上） | 選單/HUD 不需框架，a11y 與 i18n 最好控制 |
| 測試 | Vitest（單元）+ Playwright（瀏覽器煙霧/E2E） | 邏輯與實際渲染雙保險 |
| 品質工具 | ESLint + Prettier + `tsc --noEmit` | CI 級別本地閘門 |

### 4.2 目錄結構
```
Site-Project/
├── docs/                  # 本計畫書、ASSETS.md、決策記錄
├── public/assets/         # 模型/音訊（含授權清單）
├── src/
│   ├── core/              # 引擎層：GameLoop、事件匯流排、資源載入、tuning.ts
│   ├── systems/           # 遊戲系統：input / physics / character / camera /
│   │                      #   abilities / collectibles / puzzles / save / audio
│   ├── world/             # 世界：地圖資料（data-driven JSON/TS）、區域、地標、聖所
│   ├── render/            # 材質系統、palette.ts、後製管線、天空
│   ├── ui/                # 選單、HUD、設定、i18n（zh-TW / en）
│   └── main.ts
├── tests/                 # Vitest 單元 + 可達性圖測試
└── e2e/                   # Playwright
```
架構原則：**關卡資料與程式分離**（世界內容是資料，系統是程式）；系統間以事件匯流排解耦；
每個系統可獨立單元測試。

### 4.3 效能預算（機器可查）
- 60 fps（Apple Silicon 筆電基準）；同屏 draw calls < 300；視錐內三角形 < 50 萬。
- 初始載入 JS（gzip）< 1.5 MB；總資產 < 15 MB；首次可玩 < 5 秒（寬頻）。
- 手段：幾何實例化（instancing）、距離霧剔除、材質合併、資產懶載入。

### 4.4 測試策略
- **單元（Vitest）**：存檔序列化、i18n 完整性（兩語系 key 對齊）、聖所謎題狀態機、
  能力門鎖可達性圖（§2.4）、移動參數數學（coyote/緩衝計時）。
- **E2E（Playwright）**：遊戲啟動 → canvas 渲染 → 主選單鍵盤可操作 → 新遊戲 →
  模擬移動輸入 → 存檔 → 重載還原。
- **手感**：無法自動化——每個里程碑由 user 試玩裁決，是正式驗收條件之一。

### 4.5 a11y 與 i18n
中英雙語（i18n key 完整性入測試）；按鍵可重映射；選單純鍵盤可操作（focus 可見）；
「減少動態」選項（關閉相機晃動/FOV kick/粒子減量）；碑文文本字幕化、字級可調；
色彩對比：UI 文字層遵守 WCAG AA。

## 6. 製作流程（我會怎麼做）

### 6.1 角色分工
- **我（主控）**：拆解任務、派工、收貨、整合、與你溝通。原則上不自己寫大量程式碼（主控不下田）。
- **實作 subagent**：每張票附「目標與背景／可驗證的驗收條件／標準回報格式」三件套才派出。
- **驗收 subagent（fresh context）**：實作者永不自我驗證——每個交付由獨立乾淨上下文的
  驗收者實際執行 `lint / typecheck / test / e2e` 並附指令輸出證據，才算通過。
- **Codex 路由（70/30）**：程式碼審查（每里程碑跑 `codex review`）、commit 訊息、
  靜態分析走 Codex；多步編排、手感迭代、與你的往返討論留在 Claude。

### 6.2 每個里程碑的固定循環
```
開票（TaskCreate 記帳）→ 拆解+先寫驗收 → 派工實作 → 隔離驗收（附證據）
→ Codex 代碼審查 → 修訂 → git commit → 回報你 + 試玩指引 → 你的回饋 → 下一里程碑
```
- 專案會初始化 git，每個里程碑至少一個 commit，可隨時回滾。
- 你只需要做兩件事：**每個里程碑結束時試玩**（`npm run dev` 一行指令，我會給步驟）、
  對手感/美術給方向性回饋。
- 中途你隨時可以插入修改需求，我會更新票與計畫再動工。

### 6.3 誠實原則
- 查不到、做不到的一律明說並標【未查證】/【已知限制＋根因＋修復路徑】，不包裝。
- 測試綠不等於驗收過——每次收貨我會親手抽查關鍵段，防「為過審而作弊」。

## 7. 風險與應對
| 風險 | 等級 | 應對 |
|---|---|---|
| 角色模型/動畫資產不合用 | 中 | 降級為程序化「光構體」角色（§3.2），已預留設計 |
| 開放世界效能超支 | 中 | M2 起就掛效能檢測腳本，超支立即在當里程碑內處理，不後移 |
| 手感調不出來 | 低-中 | M1 硬閘門 + 調參面板即時迭代 + 參數全集中一檔 |
| Rapier WASM 與 Vite 整合坑 | 低 | 使用 `-compat` 版本（已知穩定路徑）；M0 就驗證 |
| 範圍蔓延 | 中 | 支柱三「少即是精」為裁決依據；新想法一律先進 backlog 不直接動工 |
